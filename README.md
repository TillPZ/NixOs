# NixOS Config

Flake-based NixOS + Home Manager configuration.
Pin: `nixos-26.05` (see `flake.lock`). Host: `renoir`. User: `till`.

> **Golden rule:** the source of truth is this repo, **not** `/etc/nixos`.
> Created a new file → **always `git add` first**, otherwise the flake won't see it.

## Layout

```
~/.config/nixos/
├── flake.nix                # inputs (nixpkgs, home-manager) + nixosConfigurations
├── flake.lock               # pinned commits – ALWAYS commit this
├── configuration.nix        # system config
├── hardware-configuration.nix
├── home.nix                 # Home Manager: user settings + imports
└── home/
    ├── lib/
    │   └── xilinx-box.nix   # helper: generates .ini + launchers + desktop entry per box
    ├── icons/               # logos for the desktop entries (git-tracked)
    ├── vivado_2024_2.nix    # one box = a small import call of the helper
    └── nvim.nix             # Neovim + language tooling (LSPs, formatters, compiler)
```

The distrobox `.ini` files are generated by Home Manager and live as
**read-only symlinks** under `~/.config/distrobox/`. Never edit them directly –
change the `.nix` and rebuild instead.

## Build / update the system

```bash
cd ~/.config/nixos

# apply a config change (the command you'll use most):
sudo nixos-rebuild switch --flake .#renoir

# build only, without activating (to test whether it even evaluates):
sudo nixos-rebuild build --flake .#renoir

# roll back to the previous generation (if a switch breaks something):
sudo nixos-rebuild switch --rollback
```

> After the switch, watch for `home-manager-till.service` in the output –
> that's the signal Home Manager actually wrote files.

### Update inputs (bump nixpkgs / home-manager)

```bash
cd ~/.config/nixos
nix flake update                              # rewrites flake.lock
sudo nixos-rebuild switch --flake .#renoir
git add flake.lock && git commit -m "flake update $(date +%F)"
```

Don't like the update? Restore `flake.lock` from git and rebuild:

```bash
git checkout flake.lock
sudo nixos-rebuild switch --flake .#renoir
```

## Creating distroboxes

The `.ini` is produced by `nixos-rebuild switch`. **Then** build the container.
Important: the container is named after the **section name** in the `.ini`
(e.g. `[vivado_2024_2]`), not after the file name.

```bash
# build / rebuild the container from the generated .ini:
distrobox assemble create --replace --file ~/.config/distrobox/vivado_2024_2.ini

# enter:
distrobox enter vivado_2024_2

# remove:
distrobox rm vivado_2024_2
```

### Launching tools

Launchers are on the host `PATH` and as desktop icons. They source `settings64.sh`
inside the container and then start the tool:

```bash
vivado_2024_2      # Vivado 2024.2
vitis_2024_2       # Vitis 2024.2
```

### Encapsulated home

Each box uses a dedicated container home
(`~/.local/share/distrobox-homes/<box>`) via the `home=` option, so Vivado's
dotfiles/caches don't pollute the HM-managed `/home/till`. Note: distrobox is
built for *integration*, not isolation – the host home is still mounted and
reachable inside the container. Verify the encapsulated home is active with:

```bash
distrobox enter vivado_2024_2 -- bash -c 'echo $HOME'
# expected: /home/till/.local/share/distrobox-homes/vivado_2024_2
```

If it prints `/home/till`, the container predates the `home=` setting – remove and
recreate it. License config (`XILINXD_LICENSE_FILE`) must be set inside the
container, since the encapsulated home starts empty.

## Adding a new version

1. Create the box file (via heredoc, so no bracket gets lost):

   ```bash
   cd ~/.config/nixos
   cat > home/vivado_2026_1.nix << 'INNER'
   import ./lib/xilinx-box.nix {
     slug = "2026_1";
     version = "2026.1";
     image = "ubuntu:24.04";   # check OS support per version in UG973!
     # extraInitHooks = ";...";  # e.g. libtinfo5 workaround on 24.04
   }
   INNER
   ```

2. Add it to `imports` in `home.nix`:

   ```nix
   imports = [
     ./home/vivado_2024_2.nix
     ./home/vivado_2026_1.nix
   ];
   ```

3. Track, build, create the container:

   ```bash
   git add -A
   sudo nixos-rebuild switch --flake .#renoir
   distrobox assemble create --replace --file ~/.config/distrobox/vivado_2026_1.ini
   ```

> **Ubuntu version note:** Vivado/Vitis support different OS versions per release.
> 2024.x → Ubuntu 22.04 (libtinfo5 available natively). Later releases likely
> 24.04+ (libtinfo5 dropped → symlink/package workaround via `extraInitHooks`).
> Authoritative source is UG973 for the specific version.
>
> **PetaLinux** needs its own helper (`settings.sh` instead of `settings64.sh`,
> different tool commands, different dependencies). Naming scheme: `peta_<version>`.

## Neovim (LazyVim)

Split deliberately, because LazyVim is **imperative** (lazy.nvim downloads plugins
at runtime and writes `lazy-lock.json`) while Nix is **declarative**:

- **Nix/Home Manager provides the tooling** (`home/nvim.nix`): neovim, a C compiler
  **and the `tree-sitter` CLI** (required by nvim-treesitter `main`, which
  generates parsers at runtime), and all language servers/formatters. Reproducible,
  identical on both machines.
- **LazyVim manages its own plugins.** The config lives as a *separate, mutable*
  git repo in `~/.config/nvim` (not from the Nix store, so lazy.nvim can write its
  lockfile). Reproducibility there comes from the committed `lazy-lock.json`.

Languages covered: Lua (LazyVim itself), C, Rust, Go, Python, VHDL, SystemVerilog.

### Bootstrap the config (once per machine)

```bash
mv ~/.config/nvim ~/.config/nvim.bak 2>/dev/null   # back up anything existing
git clone https://github.com/LazyVim/starter ~/.config/nvim
rm -rf ~/.config/nvim/.git
cd ~/.config/nvim && git init && git add -A && git commit -m "LazyVim base"
```

### Disable Mason (NixOS compatibility)

Mason-installed binaries fail on NixOS (dynamic linker). LSPs come from Nix on the
`PATH` instead. Create `~/.config/nvim/lua/plugins/nix.lua`:

```lua
return {
  { "williamboman/mason.nvim", enabled = false },
  { "williamboman/mason-lspconfig.nvim", enabled = false },
}
```

> With Mason disabled, anything nvim-treesitter would normally auto-install must
> come from Nix instead. The `tree-sitter` CLI is the common one – without it you
> get *"Unmet requirements for nvim-treesitter: tree-sitter (CLI)"*. It's already
> in `home/nvim.nix`. If a specific parser fails on `:TSUpdate`, add its missing
> build dependency the same way (rebuild, done).

### Enable the languages

Built-in LazyVim extras for C/Rust/Go/Python: in nvim run `:LazyExtras`, then
enable `lang.clangd`, `lang.rust`, `lang.go`, `lang.python` (binaries come from
Nix, not Mason).

VHDL + SystemVerilog have no LazyVim extra – set them up manually in
`~/.config/nvim/lua/plugins/hdl.lua`:

```lua
return {
  {
    "neovim/nvim-lspconfig",
    opts = {
      servers = {
        vhdl_ls = {},   -- VHDL-LS (rust_hdl), binary from pkgs.vhdl-ls
        verible = {},   -- verible-verilog-ls, binary from pkgs.verible
      },
    },
  },
  {
    "nvim-treesitter/nvim-treesitter",
    opts = function(_, opts)
      vim.list_extend(opts.ensure_installed, { "vhdl", "verilog" })
    end,
  },
}
```

> The Treesitter `verilog` parser also covers SystemVerilog.
> VHDL-LS wants a `vhdl_ls.toml` in the project root for cross-file analysis.
> Verible wants a `verible.filelist` for project-wide symbols. VSG (your existing
> VHDL style guide) is not an LSP – wire it in via none-ls/conform if you want
> format-on-save (it's a pip tool, not in nixpkgs).

### Plugin reproducibility across machines

After the first launch (LazyVim pulls all plugins), commit the lockfile in the
nvim repo:

```bash
cd ~/.config/nvim && git add -A && git commit -m "lock plugins"
```

On the second machine, clone that nvim repo and run `:Lazy restore` to pin the
exact same plugin versions. Two lock layers that line up: `flake.lock` for the
tooling, `lazy-lock.json` for the plugins.

## Second machine

Clone the repo, then give each host its own `hardware-configuration.nix`.
Shared parts (`home/…`) are reused; only host-specific bits differ. Because
`flake.lock` is committed, both machines run on the exact same nixpkgs commit.
